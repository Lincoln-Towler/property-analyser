# Property Analyser — Web

Next.js replacement for the Streamlit dashboard. Public read-only site; the
scoring engine is a pure TypeScript port of the Python original, verified by
golden-master parity tests.

## Pages

| Route | Contents |
|---|---|
| `/` | v3 market score + gauge, confidence interval, four pillar sub-scores, key indicator tiles, auto-generated commentary, score-history sparkline |
| `/indicators` | Config-driven card per indicator: current value, target, status, history chart with threshold line. Single-point indicators show an honest "no trend" card instead of a broken chart |
| `/cycle` | Anderson 18.6-year tracker (anchored **2008**, same as the score — Year 18/Crash as of 2026) |
| `/audit` | Per-indicator freshness, point counts, source split, staleness flags (>90 days) |
| `/api/snapshot` | Cron endpoint: computes today's score and upserts into `market_score_history` |

## Local dev

```bash
cd web
npm install
cp .env.example .env.local   # fill in the Supabase URL + anon key
npm run dev
```

Without env vars the pages render a setup notice (and `/cycle` still works —
it's pure computation).

## Tests

```bash
npm test        # 24 golden-master parity tests
npm run typecheck
```

`tests/fixtures/golden.json` was generated by `tests/generate_fixtures.py`,
which runs the ORIGINAL Python scoring functions
(`property_analysis_dashboard.py`) against synthetic SQLite datasets
(`python3 tests/generate_fixtures.py tests/fixtures/golden.json` from `web/`).
The TS engine must match those outputs exactly — including
two known quirks preserved deliberately (see the PARITY NOTE in
`src/lib/scoring/engine.ts`). If you change scoring behaviour on purpose,
regenerate fixtures and land the change as its own commit with the score
delta recorded.

## Deploy (Vercel)

1. Import the repo in Vercel, set **Root Directory** to `web/`.
2. Env vars: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,
   `SUPABASE_SERVICE_ROLE_KEY`, `CRON_SECRET` (any random string).
3. `vercel.json` schedules `/api/snapshot` daily at 21:00 UTC (~7am AEST,
   after the overnight n8n run). Vercel sends `Authorization: Bearer $CRON_SECRET`
   automatically.
4. Pages use ISR (`revalidate = 3600`): visitors get cached HTML, data
   refreshes hourly.

## Before going live — database migrations

Apply `../supabase/migrations/` in order (Supabase SQL editor or CLI):

1. `0001` — dedupe history + unique constraint + indicator-name guard
2. `0002` — combined view recreated as SECURITY INVOKER (owned here, not by app code)
3. `0003` — `market_score_history` table
4. `0004` — **read the pre-flight comments first** (verify n8n's credential),
   then enable RLS: anon becomes read-only on the four public tables,
   everything else deny-all

Take a manual `pg_dump` first — the Supabase free tier has no automated backups.
